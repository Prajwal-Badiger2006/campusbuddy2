SCREENS LIST
The application contains 19 screens organized across four distinct architectural layers. Each layer determines how
the screen is placed in the navigation stack and when it is shown to the user.
AUTH SCREENS
SplashScreen
Entry point. Checks session and routes user to the correct destination
on app launch.
LoginScreen
Allows existing users to authenticate with college email and password.
SignupScreen
New user registration with full college identity fields.
EmailVerificationScreen
6-digit OTP verification to confirm college email ownership.
ForgotPasswordScreen
Sends a password reset link to the registered college email.
SETUP SCREENS
ProfileSetupScreen
Shown once after email verification. User selects interests, skills,
availability, and goals via chip UI.
OnboardingScreen
Shown once after profile setup. Introduces the app features to the new
user.
MAIN SCREENS (BottomNavigation)
HomeScreen — Tab 1
Dashboard with greeting, streak badge, reliability badge,
recommended partners, and open requests.
RequestsScreen — Tab 2
Two-tab screen: Browse Requests (all college requests) and My
Requests (creator view).
ChatsScreen — Tab 3
List of all active conversations with real-time unread badge on
BottomNav tab.
ProfileScreen — Tab 4
Full user profile with streak card, reliability card, edit, settings, and
logout.

Form to post a new partner request with type, title, range, availability,
and needed count.

Responder view of a request with creator card and Request to Join
button.

Creator view of their own request with accept/reject controls per
response.
PartnerProfileScreen
Full profile of another student with compatibility card and optional chat
button.
ChatScreen
Real-time 1:1 or group chat screen with message bubbles and send

DATA MODELS (ENTITIES / STATES)
Eight Kotlin data classes define the complete data layer of CampusBuddy. Each model maps directly to a database
table. Below are the full field definitions with types, nullability, defaults, and constraints.
UserProfile
data class UserProfile(
val id: String // Unique user identifier (auth UID)
val fullName: String // User's full real name
val collegeEmail: String // Must be college-domain email
val registrationNumber: String // Unique per college, validated on signup
val collegeName: String // College name string
val department: String // e.g. Computer Science, Mechanical
val year: String // e.g. 1st, 2nd, 3rd, 4th
val bio: String? // Optional short bio
val profilePhotoUrl: String? // Optional cloud storage URL
val interests: List // e.g. ["DSA", "ML", "Photography"]
val skills: List // e.g. ["Kotlin", "Python", "Figma"]
val availability: List // e.g. ["morning", "evening", "weekends"]
val goals: List // e.g. ["Crack Placements", "Win Hackathons"]
val isVerifiedStudent: Boolean // default: false → true after OTP
val status: UserStatus // UNVERIFIED | VERIFIED | ALUMNI | BANNED
val role: UserRole // USER | ADMIN
val currentStreak: Int // default: 0, increments on daily login
val reliabilityScore: Float // default: 0.0, recalculated on match/msg
val createdAt: Long // Unix timestamp of account creation
)
PartnerRequest
data class PartnerRequest(
val id: Long // Auto-generated primary key
val creatorId: String // FK → UserProfile.id
val type: RequestType // STUDY|PROJECT|HACKATHON|EVENT|PLACEMENT
val title: String // Min 5 chars, not empty
val description: String? // Optional extended description
val subjectOrTopic: String? // Optional subject/topic label
val preferredRange: RangeType // CLASS|DEPARTMENT|COLLEGE|NEARBY|ONLINE
val preferredSkills: List // Optional skill filter chips
val availability: String? // Optional availability string
val neededCount: Int // 1–10, how many partners needed
val acceptedCount: Int // default: 0, increments on each accept
val status: RequestStatus // OPEN|FULL|CLOSED|CANCELLED
val createdAt: Long // Unix timestamp
)

03
ENUM / STATE VALUES
Seven enumerations define every possible state across the application. These are used for type safety, UI state
management, conditional logic, and database persistence.
enum class UserStatus
UNVERIFIED
User has created an account but has not yet completed email OTP
verification. Cannot post requests or message others.
VERIFIED
Confirmed college student. Full access to all app features.
ALUMNI
User has graduated or left college. May have limited access.
BANNED
Admin-banned account. Redirected to LoginScreen with suspension
message on app launch.
enum class UserRole
USER
Normal student account with standard permissions.
ADMIN
Admin account. Can moderate reports, ban users, and view all
content.
enum class RequestType
STUDY
Request to find a study partner for exams or coursework.
PROJECT
Request to find teammates for an academic or personal project.
HACKATHON
Request to form a hackathon team.
EVENT
Request to find a buddy for college events.
PLACEMENT
Request to find a partner for placement/interview preparation.
enum class RangeType
CLASS
Visible only to students in the same class section.
DEPARTMENT
Visible to entire department.
COLLEGE
Visible to the entire college.
NEARBY
Visible to students in nearby colleges.
ONLINE
No location restriction; visible to all verified users.

enum class RequestStatus
OPEN
Request is active and accepting new responses.
FULL
acceptedCount has reached neededCount. No more responses
accepted. closeRequestButton hides.
CLOSED
Creator manually closed the request. No longer accepting responses.
CANCELLED
Creator cancelled the request before completion.
enum class ResponseStatus
PENDING
Response submitted, awaiting creator's accept or reject decision.
ACCEPTED
Creator accepted this response. Match + Conversation created.
REQUEST_ACCEPTED notification sent.
REJECTED
Creator rejected this response. REQUEST_REJECTED notification
sent to responder.
enum class NotificationType
REQUEST_RECEIVED
Sent to request creator when a new response is submitted. Icon:
Person. Routes to MyRequestDetailsScreen.
REQUEST_ACCEPTED
Sent to responder when creator accepts. Icon: Checkmark. Routes to
ChatScreen.
REQUEST_REJECTED
Sent to responder when creator rejects. Icon: X. Routes to
RequestsScreen (My Requests tab).
NEW_MESSAGE
Sent to recipient when a new chat message is received. Icon: Chat.
Routes to ChatScreen.
STREAK_MILESTONE
Sent to user when streak hits 7, 14, or 30 days. Icon: Fire. Routes to
ProfileScreen.

MY REQUEST DETAILS SCREEN — CREATOR VIEW
Screen
MyRequestDetailsScreen
Trigger
Creator taps one of their own requests in My Requests tab
Components
Request info · statusBadge · acceptedCountText · closeRequestButton ·
responsesSection
responsesSection
List of all PartnerResponse rows for this request, each showing responder info,

Accept Response Logic
I Creator taps Accept → confirmation dialog appears.
I If confirmed: acceptButton enters loading state.
I After DB save: PartnerResponse.status → ACCEPTED.
I Buttons on that row disappear. Green "Accepted" label appears.
I acceptedCountText increments.
I A Match row is created in DB (user1Id=creatorId, user2Id=responderId, requestId).
I A Conversation is created in DB with both users as members.
I Notification (REQUEST_ACCEPTED) pushed to responder.
I If acceptedCount == neededCount: PartnerRequest.status → FULL. closeRequestButton hides. Open Chat button
appears on that response row.
Reject Response Logic
I Creator taps Reject → no confirmation required.
I PartnerResponse.status → REJECTED.
I Buttons disappear. Red "Rejected" label appears.
I Notification (REQUEST_REJECTED) dispatched to responder.
Close Request Logic
I Creator taps closeRequestButton (only visible when status=OPEN) → confirmation dialog.
I If confirmed: PartnerRequest.status → CLOSED.
I statusBadge updates to grey. closeRequestButton hides.
Data Flow
Updates PartnerResponse.status · Writes Match row · Writes Conversation row ·
Creates notification logs

I ActivityLog row inserted: activityType=MESSAGE, activityDate=today.

STREAKS & RELIABILITY DATA FLOW
Streak Calculation Logic
I A streak increments by 1 each day the user performs at least one transaction (login, message, match, or session).
I Streak calculation runs on every login event.
I Query ActivityLog for any entry where userId==currentUserId AND activityDate == yesterday.
I Entry found → UserProfile.currentStreak += 1
I No entry found → UserProfile.currentStreak = 1 (reset)
I Write new ActivityLog row: activityType=LOGIN, activityDate=today.
Streak Milestones
7 Days
Trigger STREAK_MILESTONE notification: title = "7-day streak!", fire
icon, routes to ProfileScreen
14 Days
Trigger STREAK_MILESTONE notification: title = "14-day streak!"
30 Days
Trigger STREAK_MILESTONE notification: title = "30-day streak!"
Reliability Score Calculation
The reliability score measures how often a user follows through after being matched. It is recalculated dynamically and
persisted to UserProfile.reliabilityScore.
totalInteractions = Count(Match rows where user1Id OR user2Id == currentUserId)
completedInteractions = Count(Match rows with at least 1 Message exchange in linked Conversation)
reliabilityScore = (completedInteractions / totalInteractions) x 100
I Recalculated in the repository layer when a new Match row is created.
I Recalculated in the repository layer when a new Message row is successfully committed.
I Result persisted immediately back to UserProfile.reliabilityScore.
I Displayed on ProfileScreen reliabilityCard and on partner cards in HomeScreen.

GENERAL UI RULES FOR ALL SCREENS
These rules apply universally across every screen in the application and must be implemented without exception.
Loading States
I Every screen that loads async data must display polished grey skeleton placeholder cards/blocks.
I Blank white or empty screens during loading are strictly forbidden.
I Skeleton layout must mirror the shape and structure of the real content it represents.
Empty States
I Any list with zero records must display an illustrative image (non-default, contextually relevant).
I Accompanying helper text must be contextually precise and actionable (e.g. "No open requests found. Create one!").
I Empty states must not show a plain 'No data' text without visual treatment.
Error Handling
I Network failures and database errors must trigger a Snackbar alert immediately.
I Snackbar message must specify what failed (not a generic error).
I Snackbar must include an active Retry action button.
Button Interaction Controls
I Any button that triggers a remote operation must disable its tap listener on tap.
I A circular progress indicator must render inside or near the button until the transaction completes.
I On success or failure, button must return to interactive state (enabled or error state).
Confirmation Overlays
I Destructive or irreversible workflows must require a confirmation modal.
I Affected flows: Logout, Delete Account, Close Request.
I Modal must contain exactly two clearly labeled actions: Cancel and Confirm.
I No destructive action should execute without an explicit Confirm tap.
Navigation Standards
I All stack-pushed secondary screens must display an explicit back arrow in the top bar.
I Back arrow must route cleanly back to the calling screen without side effects.
I Disabled UI controls must be visually greyed out.
I Disabled controls must display clarifying tooltip text or inline hint text explaining why they are disabled.
I Disabled controls must NEVER show generic system exception bubbles or crash dialogs.
